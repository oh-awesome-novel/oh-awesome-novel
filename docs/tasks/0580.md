# 0580 Git History And Sync Page

> Status: Planned
> Milestone: M8 Minimal Copilot Interface

## Goal

在 workspace 工作台中增加 Git 记录查看和操作页面，让作者能看到小说工程的历史、当前未提交变更、同步状态，并能执行明确的 Git 操作。

优先寻找并复用轻量级开源 Git GUI / Git history UI，不要从零实现完整 Git GUI。如果没有合适的轻量级方案，则实现最小自研页面：查看 `git log`、查看当前状态、自动同步、PendingAction diff view，以及 PendingAction 执行后按配置自动提交一次 Git。

同时，workspace 顶部 toolbar 需要提供“在外部编辑器中打开工作空间”的入口，让用户可以直接用 VS Code、Zed、WebStorm / JetBrains IDE 等专业编辑器打开当前 workspace root，并使用这些编辑器内置的 Git 管理能力。

## Open Source First

实现前必须先调研可复用的轻量级开源方案，并记录结论。

候选方向：

- 轻量 Git history / Git log UI component。
- 可嵌入的 Git diff / history viewer。
- [`simple-git`](https://www.npmjs.com/package/simple-git) / [`steveukx/git-js`](https://github.com/steveukx/git-js)，用于调用应用内自带的 Git 二进制程序。
- `isomorphic-git` 这类 JS Git 操作库，只作为备选调研方向。

已知注意事项：

- `simple-git` 可以作为默认 Git 操作层：`steveukx/git-js` 将其定位为 Node.js 应用中运行 Git 命令的轻量接口。它不提供 GUI，但能以 Node API 方式调用 Git 命令，适合 Electron backend / main process 封装。
- 应用需要自带 Git 二进制程序，并让 `simple-git` 显式使用该 binary，避免依赖用户系统是否已安装 Git。
- `isomorphic-git` 是 JavaScript Git 实现，可在 Node 和浏览器环境读写 Git 仓库，并支持 fetch / push 等操作；它更适合作为 Git 操作库，不是完整 GUI。
- `gitgraph.js` 可绘制 Git 图，但项目已归档且长期未维护，不应作为默认方案，除非后续 fork 或替代项目满足维护性要求。
- Gitea / Forgejo / GitLab 这类 forge 太重，不适合作为内嵌 workspace Git GUI。

如果选择第三方开源项目，必须评估：

- license 是否兼容本项目。
- 是否仍在维护。
- 是否足够轻量，可嵌入 Electron / Vue 工作台。
- 是否允许只打开本地 workspace 仓库。
- 是否不会引入服务端数据库、账号系统或代码托管平台。

## Git Operation Layer

Git 操作层第一选择使用 [`simple-git`](https://www.npmjs.com/package/simple-git)，源码仓库为 [`steveukx/git-js`](https://github.com/steveukx/git-js)。

要求：

- `simple-git` 只能在 backend / Electron main / Git integration 层使用。
- Vue frontend 不直接调用 Git，也不直接访问 filesystem。
- 应用打包时自带 Git 二进制程序。
- Git integration 层必须显式配置 `simple-git` 使用应用内自带 Git binary。
- 优先使用 `simple-git` 的 `binary` / `customBinary` 配置能力绑定应用内 Git binary。
- 所有 Git 命令的 working directory 必须锁定到当前 workspace root。
- Git 命令不得接受 frontend 传入的任意 shell 字符串。
- Git 操作结果以结构化数据返回 UI。

第一版需要封装的 Git 操作：

- status。
- log。
- show commit / commit diff。
- add touched files。
- commit。
- fetch。
- pull fast-forward only。
- push。

如果打包 Git binary 在某个平台不可用，UI 应显示 Git 不可用状态，并禁止 commit / sync 操作。

## Fallback Minimal Scope

如果没有合适的轻量级开源 Git GUI，则第一版只实现：

- Git log 列表。
- 当前分支 / commit 信息。
- 当前 workspace dirty 状态。
- 单个 commit 详情。
- Commit touched files。
- Commit diff preview。
- 手动同步按钮。
- 自动同步状态展示。
- PendingAction accept 后按配置自动提交一次 Git。
- 关闭自动提交时的快捷提交按钮。
- PendingAction diff view。

第一版不需要实现：

- 分支管理 UI。
- 复杂 rebase / merge / cherry-pick。
- 冲突解决器。
- 类似 SourceTree / GitKraken 的完整 Git GUI。

## External Editor Entry

为了降低内置 Git UI 的复杂度，workspace 顶部 toolbar 需要支持把当前 workspace root 打开到用户已安装的专业编辑器。

第一版支持的目标：

- VS Code。
- Zed。
- WebStorm / JetBrains IDE。

交互要求：

- toolbar 提供一个明确的“外部编辑器”入口，可以是按钮、下拉菜单或命令菜单。
- 用户选择编辑器后，应用打开当前 workspace root，而不是打开单个文件。
- 如果目标编辑器未安装或无法打开，UI 显示可恢复错误，不改变 workspace 状态。
- 如果多个 JetBrains IDE 可用，第一版可以优先支持 WebStorm；后续再扩展为 JetBrains IDE 列表。
- 该入口只负责启动外部编辑器，不在本应用内执行 Git commit、push、pull 等操作。

安全和边界要求：

- Frontend 只能传递结构化 editor id，例如 `vscode`、`zed`、`webstorm`。
- Backend / Electron main 层必须使用 allowlist adapter 打开外部编辑器。
- 不接受 frontend 传入任意可执行文件路径、shell 命令或命令行参数。
- 打开的路径必须是当前 workspace root，且经过 workspace boundary 校验。
- 外部编辑器中的 Git 操作属于用户显式离开本应用后的手动行为，不属于 PendingAction 自动提交流程。
- 本应用仍必须保留 Human Approval、PendingAction diff view 和 Git history 页面；外部编辑器入口不能替代这些内部确认链路。

## PendingAction Auto Commit Configuration

新需求：PendingAction 执行成功后，默认需要自动提交一次 Git，但用户必须可以通过配置关闭该行为。

第一版 workspace 配置：

```yaml
git:
  autoCommitOnAccept: true
```

默认值为 `true`。

流程：

```text
PendingAction accepted
    ↓
materialize shadow write
    ↓
git diff visible
    ↓
validate changed files
    ↓
git.autoCommitOnAccept?
    ↓ true
git add touched files
    ↓
git commit with generated message
    ↓
refresh Git history page
    ↓
attempt auto sync if configured
    ↓ false
show dirty state and quick commit button
```

Commit message 第一版可使用确定性模板：

```text
chore(novel): apply pending action <short-id>

<PendingAction title>
```

后续可以让 AI 建议 commit message，但不能在没有人类确认 PendingAction 的情况下提交。

当 `git.autoCommitOnAccept` 为 `false` 时：

- PendingAction accept 仍然 materialize shadow write。
- 不自动 `git add`。
- 不自动 `git commit`。
- 不触发自动同步。
- UI 必须展示当前 dirty 状态。
- UI 必须提供“Commit now”快捷按钮，让用户在任何想提交的时候提交。

快捷提交要求：

- 可以放在 workspace toolbar、Git 页面、PendingAction accept 成功后的状态提示中。
- 点击后展示将提交的 dirty files 和 diff。
- 用户必须能确认或编辑 commit message。
- 提交成功后刷新 Git status / Git history。
- 提交失败时展示错误并保留 dirty 状态。

## PendingAction Diff View

默认文件操作会先写入 `workspace/.workspace` shadow recovery 路径。用户 Accept 前，真实目标文件不得被覆盖。

Git 页面或 PendingAction 面板需要提供 diff view，用于比较：

```text
左侧：真实目标文件当前内容
右侧：.workspace shadow write 中的候选内容
```

该 diff view 展示 PendingAction 即将造成的变化，而不是已经写入 Git working tree 后的普通 `git diff`。

Diff view 要求：

- 支持按 PendingAction 查看。
- 支持按 touched file 查看。
- 显示真实目标路径。
- 显示 shadow 文件路径或恢复引用。
- 对新增文件、修改文件、删除文件分别给出清晰状态。
- Accept 前只读取 `.workspace` shadow 内容和真实文件内容。
- Reject 后 diff view 应显示该 PendingAction 已废弃或不可用。
- Accept 后 diff view 可以切换为普通 Git diff / commit diff。

这条链路用于让用户在 Human Approval 前明确看到 `.workspace` 内候选内容与真实 workspace 文件的区别。

## Auto Sync

自动同步用于把本地 Git 历史同步到远端。第一版可以只支持当前 Git remote 的简单同步。

建议策略：

- PendingAction 自动提交成功后触发一次 sync。
- 应用启动或进入 workspace 后可以检查 remote 状态。
- 用户可以手动点击同步。
- 同步失败只展示错误，不回滚已完成的本地 commit。

第一版 sync 能力：

- 显示 remote 名称和当前分支。
- `fetch` / `pull --ff-only` 或等价安全策略。
- `push` 当前分支。
- 失败原因展示。
- 网络不可用、未配置 remote、鉴权失败时可恢复。

- Git status summary。
- Git log list。
- Commit detail view。
- Diff preview。

- Manual sync action。
- Auto sync status.

## Done Criteria

- 用户可以从 workspace toolbar 或设置入口打开 Git 页面。
- 用户可以从 workspace toolbar 选择 VS Code、Zed、WebStorm / JetBrains IDE 打开当前 workspace root。
- 外部编辑器入口不会接受任意 shell 命令或任意 executable path。
- 外部编辑器未安装或启动失败时，UI 会展示错误，并且不改变 workspace 内容。
- 页面可以展示当前 workspace 的 Git log。
- 页面可以展示当前 dirty 状态。
- 页面可以查看 commit 的 touched files 和 diff。
- PendingAction accept 前，用户可以查看 `.workspace` shadow 内容与真实目标文件的 diff。
- PendingAction diff view 可以按 touched file 切换。
- PendingAction diff view 不会 materialize shadow write。
- 默认配置下，PendingAction accept 成功后会自动创建一个 Git commit。
- 用户可以通过 workspace 配置关闭 PendingAction accept 后自动 commit。
- 关闭自动 commit 后，PendingAction accept 只 materialize 文件并展示 dirty 状态。
- 关闭自动 commit 后，用户可以通过快捷按钮提交当前 dirty 变更。
- PendingAction reject 不创建 commit。
- 自动提交只包含 PendingAction touched files。
- 自动提交失败时展示错误，并保留已写入文件和可见 dirty 状态。
- 快捷提交会展示将提交的文件和 diff，并要求用户确认 commit message。
- 配置 remote 后可以手动同步。
- 配置 remote 后 PendingAction 自动提交成功可触发自动同步。
- 关闭自动 commit 时，PendingAction accept 不触发自动同步。
- 同步失败不隐藏本地 commit。

## Constraints

- 不自己实现完整 Git GUI。
- 优先复用轻量级开源项目；无合适项目时只实现 fallback minimal scope。
- Git 操作层默认使用 `simple-git` 调用应用内自带 Git binary。
- 不引入 Gitea / Forgejo / GitLab 这类完整 forge。
- 不引入服务端数据库。
- 不绕过 Human Approval：只有 accepted PendingAction 可以触发写入和自动 commit。
- Tool 不直接 commit；commit 应发生在 Human Approval / Git integration 层。
- 不允许自动提交未被 PendingAction touchedFiles 覆盖的文件。
- `git.autoCommitOnAccept: false` 时不得自动提交或自动同步。
- 快捷提交是用户显式 Git 操作，不属于 AI 自主行为。
- 外部编辑器入口只能打开当前 workspace root，不能作为任意命令执行器。
- 外部编辑器中的 Git 操作是用户手动操作，不应被记录为 PendingAction 自动提交。
- Accept 前 diff view 只能比较真实文件与 `.workspace` shadow 内容，不能提前写入真实目标。
- Frontend 不能直接执行 Git 命令或 filesystem 读写。
- 不在同步失败时静默丢弃本地变更。
