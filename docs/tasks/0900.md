# 0900 Project References And Deconstruction

> Status: Needs Review
> Milestone: Final Reference Material Layer

## Goal

最后实现项目参考功能。`examples/` 用来存放项目参考材料，而不是实际小说正文 workspace。

本任务现在明确为 **Reference Work Deconstruction / 参考作品拆解层**：

- 可以导入用户提供的参考小说、设定集、样章或风格样本。
- 保留原始源文件和来源元数据。
- 通过拆解 pipeline 提炼出可查询的结构化创作参考。
- 写作 agent 默认只读取 distilled reference，不直接吞原文大文件。
- 参考材料只作为 inspiration / technique / benchmark，不作为当前小说 truth files。

这类似 `oh-story-claudecode` 的 `story-long-analyze` / `story-import` / `对标` 思路，但必须按 OAN 的 filesystem-first、PendingAction、Human Approval 和 Object File Tree 边界重写。

## Existing 0900 Draft To Preserve

旧版 `0900.md` 已经有三个正确判断，必须保留：

- `examples/` 不是小说正文 workspace。
- 原始 source 和 distilled reference 要分开。
- reference 不应成为隐藏事实源，也不应默认塞进 prompt。

需要修改的是：旧版结构太平，只定义了 `examples/sources/` 和 `examples/distilled/`，不足以承载多本参考作品、来源元数据、拆解进度、质量门、上下文选择和 adoption 边界。

## Proposed Storage Scope

旧草案：

```text
examples/
├── sources/
└── distilled/
```

建议升级为每个参考作品一个 bundle：

```text
examples/
├── README.md
├── references.yaml
└── references/
    └── <reference-id>/
        ├── metadata.yaml
        ├── sources/
        │   ├── original.<ext>
        │   └── source-manifest.yaml
        ├── progress.yaml
        ├── deconstruction/
        │   ├── quick-preview.md
        │   ├── chapters/
        │   │   ├── 0001-summary.md
        │   │   └── 0001-deep-dive.md
        │   ├── plotlines.md
        │   ├── characters.md
        │   ├── relationships.md
        │   ├── worldbuilding.md
        │   ├── timeline.md
        │   ├── tropes.md
        │   └── style-profile.md
        ├── distilled/
        │   ├── writing-style.md
        │   ├── pacing.md
        │   ├── hooks.md
        │   ├── scene-techniques.md
        │   ├── character-techniques.md
        │   └── do-not-copy.md
        └── context/
            ├── index.yaml
            └── reference-summary.md
```

说明：

- `sources/` 保存用户显式导入的原始材料和 checksum，不作为默认 context。
- `deconstruction/` 保存拆解过程产物，强调可追溯和可复核。
- `distilled/` 保存写作时真正推荐读取的参考摘要。
- `context/` 保存面向 agent 的轻量 index / summary，是 reference context selector 的主要入口。
- `references.yaml` 是项目级参考作品索引，只记录 id、标题、类型、启用状态、distilled summary 路径等。

## Learnings From oh-story-claudecode

### 1. Analyze Before Use

`oh-story-claudecode` 的 `story-import` 不把已有文本直接当成写作项目，而是先分析再迁移。

OAN 应吸收为：

- 参考作品必须先生成 deconstruction report。
- 写作 agent 读取的是拆解后的 technique / rhythm / style / trope，而不是直接把原文塞入 prompt。
- 如果用户想把“自己的旧稿”导入为当前小说 workspace，应走另一个 manuscript import / adoption flow，不能和 reference benchmark 混在一起。

### 2. Keep Original Source But Do Not Prompt With It By Default

`story-long-analyze` 会备份原文，并基于原文切章、摘要、聚合。

OAN 应吸收为：

- 保存源文件和 checksum，方便用户复查。
- 默认 agent context 只读取 `context/reference-summary.md` 或 `distilled/*`。
- 原文只在用户显式要求“查看来源 / 重新拆解 / 校验引用”时读取。
- 任意从原文抽取的事实或技巧都要带 source pointer，例如 chapter id、section id、chunk id。

### 3. Stage Pipeline With Resume

`oh-story-claudecode` 的拆文有 Stage 0-6：概要、黄金三章、逐章摘要、聚合分析、设定关系、汇总报告、文风。

OAN 可以改造成更轻的阶段：

1. **Import Source**：复制或登记源文件，写 `metadata.yaml` / `source-manifest.yaml`。
2. **Detect Structure**：识别章节边界、标题、字数、可能的体裁。
3. **Quick Preview**：先拆开篇或前 1-3 章，生成 `quick-preview.md`，让用户确认是否继续。
4. **Chapter Summaries**：分块 / 分章生成摘要和关键技法观察。
5. **Aggregate Deconstruction**：聚合 plotlines、characters、relationships、worldbuilding、tropes。
6. **Style Profile**：生成文风、句式、节奏、对话、场景技法摘要。
7. **Distill For OAN**：生成写作时可用的 distilled reference 和 context index。

长任务必须有 `progress.yaml`，支持失败记录、断点恢复和重跑某一阶段。

### 4. Quick Preview Gate

`oh-story-claudecode` 在黄金三章后会停靠，让用户决定是否继续全量拆解。

OAN 应吸收为：

- 大文本导入后先做 quick preview，不默认跑完整拆解。
- Preview 应说明这份参考适合学什么、不适合学什么。
- 用户确认后才进入全量拆解，避免对不合适材料浪费 token。

### 5. Style Recall Must Be Narrow

`oh-story-claudecode` 日更时会读取 `文风.md` 和匹配章节摘要 / 深度拆解，而不是全量读取对标书。

OAN 应吸收为 reference context selector：

- 输入当前写作任务、题材、章节目标、情绪、场景类型。
- 从 enabled references 中选择少量 distilled entries。
- 输出 included / omitted / reason / token budget。
- 只返回 technique notes，不返回可复制正文。

### 6. Deconstruction Quality Gates

`oh-story-claudecode` 对章节数、摘要数量、角色合并、设定事实可追溯等有质量门。

OAN 应吸收为：

- 检查识别到的章节数与摘要文件数一致。
- 聚合角色 / 设定时记录 confidence 和 source pointers。
- 不确定的内容写入 `uncertain.md` 或待确认区，不硬编成事实。
- 如果原文缺失或章节边界不可靠，必须在 report 中标记。

## Additional Learnings From Existing References

### StoryForge Import Pipeline

StoryForge 的导入 pipeline 提醒 OAN：

- 大文本导入要 chunk + rolling context，不能一次性吞全文。
- 每 N 块做一次合并去重。
- 导入结果先写 shadow / pending / report，用户确认后再进入正式对象。

对 0900 来说，这意味着 reference deconstruction 也应支持 chunked processing 和 resumable progress。

### InkOS Import / Style Workflow

InkOS 提醒 OAN：

- 风格分析应成为可审阅文件，例如 `style-profile.md`。
- 续写或参考前应生成“当前连续性 / 风格理解报告”，让作者确认 agent 理解没跑偏。

对 0900 来说，这意味着 reference style 不能只是内存 prompt，应落成可读、可复查、可禁用的文件。

## OAN-Specific Changes Needed

### 1. Replace Flat Distilled Folder With Reference Bundles

旧版 `examples/distilled/outlines`、`writing-style`、`worldbuilding`、`characters` 适合单参考作品，不适合多参考作品。

需要改为：

- 每个 reference 独立目录。
- 项目级 index 指向各 reference。
- UI / agent 可以按 reference 启用、禁用、选择。

### 2. Separate External Reference From Own-Manuscript Import

参考作品拆解和导入自己的旧稿不是同一件事。

- External reference：只能作为技法、节奏、题材、风格参考。
- Own manuscript import：可以生成当前 workspace 的 summary、state、timeline、foreshadow、style profile 候选。

0900 只负责 external reference layer。自己的旧稿导入应属于 import workspace / manuscript migration workflow。

### 3. Add Provenance And Rights Metadata

每个 reference 必须记录：

- source type：novel / chapter sample / style sample / setting bible / notes。
- user-declared rights：owned / public domain / licensed / excerpt / unknown。
- source path / import date / checksum。
- allowed usage：analysis only / style inspiration / structure reference / no direct quotation。

OAN 不应帮助用户复制参考作品正文、桥段或可识别表达。拆解输出应是转化性结构分析和写作技法摘要。

### 4. Make Reference Context Explicit

写作时不能有隐藏 reference。

Agent 每次使用 reference 时应说明：

- 使用了哪个 reference id。
- 读取了哪个 distilled file。
- 为什么与当前章节相关。
- 哪些 reference 被排除。
- 是否读取了原文；默认应为否。

### 5. Add No-Copy Guardrails

Distilled output 应包含 `do-not-copy.md`：

- 不照搬角色名、地名、专有设定。
- 不复用桥段排列。
- 不复用可识别句子、段落、对白。
- 只学习结构、节奏、情绪机制、场景功能和抽象技法。

### 6. Adoption Is Always Explicit

参考拆解结果不能自动覆盖 OAN truth files。

允许的 adoption：

- 把“可借鉴技法”加入 `.oan/constitution/style.md` 的候选 PendingAction。
- 把“题材节奏偏好”加入 workflow / skill override 的候选 PendingAction。
- 把“场景技法”作为一次 `/规划下一章` 的参考输入。

禁止的 adoption：

- 直接把参考作品角色设定写入 `characters/`。
- 直接把参考作品世界规则写入 `world/`。
- 直接把参考作品时间线 / 伏笔写入当前小说 truth files。

## Deliverables

- Reference source import with provenance metadata。
- Reference bundle index under `examples/references.yaml`。
- Source manifest with checksum and chapter boundary metadata。
- Quick preview deconstruction for first 1-3 chapters or selected excerpt。
- Resumable full deconstruction pipeline。
- Chapter summaries and optional deep dives。
- Aggregate plotline / pacing / trope / character / worldbuilding notes。
- Style profile and scene technique summary。
- Distilled reference context files。
- Reference context selector。
- UI or command surface for enabling / disabling references per writing session。
- PendingAction-based adoption flow for any reference-derived changes to current workspace facts or constitution。

## Done Criteria

- `examples/` 不再被当作实际小说 workspace。
- 每个参考作品都有独立 bundle、metadata、source manifest 和 distilled summary。
- 原始 source 文件不会默认进入 agent prompt。
- AI 可以读取 distilled reference summary，而不是默认读取原始大文件。
- Reference context selector 会输出 included / omitted / reason。
- 参考材料不会自动覆盖 workspace 中的事实源。
- 用户可以选择是否把某个 reference 加入当前 agent context。
- 用户可以禁用某个 reference，禁用后 agent 不再读取它。
- 对大文本拆解支持 progress / resume / failure report。
- 对 reference-derived adoption，必须生成 PendingAction / diff，并由用户确认。

- reference-derived adoption 到 truth files 仍需专门 PendingAction flow。
- `examples/` 下的 reference 文件是用户显式导入材料，不是当前小说 truth files。

## Constraints

- 最后实现，不阻塞核心 agent loop、workspace、session、写入和 Apply Engine。
- 不使用 reference 作为隐藏事实源。
- 不把 reference source 原文默认塞进 prompt。
- 不引入 vector DB 作为 reference truth source。
- 不默认联网下载参考作品。
- 不复制 `oh-story-claudecode` 的目录、prompt 或实现；只吸收 pipeline 思路。
- 不把拆文 / 对标变成每章写作默认必跑步骤。
- 不绕过 OAN 的 Human Approval 与 Git diff 工作流。

## Open Design Questions

- `examples/` 是否最终应保留这个名称，还是改为更明确的 `references/`？旧任务已固定 `examples/`，本文暂保留。
- 原文 source 是否允许存入 workspace Git，还是默认加入 `.gitignore` 并只保留 manifest？这取决于用户材料版权和隐私策略。
- reference deconstruction 是否应作为 core tool、extension workflow，还是 project-local skill？
- Quick preview 的默认范围是前三章、用户选段，还是按字数阈值自动决定？
- 引用少量原文作为 style anchor 是否允许？如果允许，需要严格限制长度并默认关闭。
