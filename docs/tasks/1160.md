## Goal

让 `playerUnknown` 世界事件可以先在 selected branch 发生，随后通过新的、可感知的 `informationSpread` 事件显式揭示为 `rumor` 或 `playerVisible`；知识投影随既有 turn snapshot、Restore 与 Retry 恢复，且绝不原地改写 hidden event。

## Scope

- reserved `playKnowledge` branch-local state schema、严格 normalization 与 append-only transition。
- settlement `knowledgeChanges` typed protocol，以及与当前回合 `informationSpread` event 的一一 evidence 约束。
- 只允许 `playerUnknown -> rumor | playerVisible` 与 `rumor -> playerVisible`；拒绝降级、重复、同回合、自引用、未知或 sibling event。
- world-referee context 只提供当前 selected ancestor 上可揭示的有界候选与现有 projection。
- Player / Author reveal projection：Player 只看到当前安全揭示文本与通用因果说明，Author 可查看完整 hidden source chain。
- v4 Immersive Journey、v5 Scene Rehearsal、Restore、Retry、Backend、strict Client 与 Desktop 集成。

## Out Of Scope

- 角色级 `grantKnowledge`、Scene Memory、F4 advanced Director controls。
- 修改旧事件 visibility、向 Player 返回 hidden event 的原始 title / summary / cause。
- 新 endpoint、平行 knowledge 文件、数据库、后台自治世界或 canonical 写入。

## Done Criteria

- knowledge record 只引用当前 artifact 的严格祖先事件；同回合、未知、discarded 或 sibling 事件 fail closed。
- 每个 reveal change 恰好绑定当前 settlement 的一个 `informationSpread` event，且 event cause 显式引用被揭示事件。
- hidden event 本体保持 byte-equivalent；Player projection 不返回其 id、title、summary、reason 或隐含 fallback 文本。
- Restore / Retry 后 knowledge state 与 selected snapshot 一致，旧分支的 reveal 不会污染新分支。
- raw `stateDelta.playKnowledge` 不能绕过 typed protocol；transcript append 不能修改 knowledge。
- Core、Backend、Client、Desktop 测试和 Desktop production build 通过。

## Implementation Notes

- 新增 `packages/core/src/play-knowledge.ts`，把 `playKnowledge` 固定为现有 `playLocalState` / turn snapshot 中的 v1 reserved state；记录只追加、不建立平行 ledger。
- world-referee settlement 新增 typed `knowledgeChanges`。宿主只在当前回合存在唯一、同目标可见性的 `informationSpread` event 且其 cause 引用严格 selected ancestor 时生成 record；provider 直接写 `stateDelta.playKnowledge` 会被拒绝。
- `play-session-facts.ts` 会沿每个 artifact 的真实 parent chain 重放 knowledge prefix、宿主 id、揭示事件与 visibility；Restore / Retry 继续复用既有 snapshot 语义，因此 sibling / discarded branch 不会串用知识。
- v5 Scene Rehearsal 的 provisional contribution 会过滤已用 subject、拒绝重复或不成对 reveal，Finish 聚合后仍进入与 v4 相同的 Core settlement invariant。
- strict Client 覆盖 v4 / v5 session、artifact、selected head、reserved visibility 与 contribution；Desktop reveal chain 在 Player 视角只显示通用因果说明，Author 才能查看 hidden source，且 `playKnowledge` 永不进入通用 State HUD。
- 验证覆盖 unknown → rumor → visible、旧 hidden event 不变、unknown / same-turn / sibling / downgrade / no-op / duplicate / ambiguous 拒绝、raw reserved write、transcript append、Restore / Retry 隔离，以及 Author → Player 同步 scrub。
- 当前合并回归结果：Core 28 files / 231 tests、Backend 4 / 69、Client 5 / 91、Desktop 34 / 139 全部通过；Desktop production build 通过，仅保留既有 chunk-size warning。

## Constraints

- `playKnowledge` 是既有 `play-local-state.yaml` 与 turn snapshot 中的 reserved state，不建立第二事实源。
- knowledge transition 由宿主从 typed change 生成；provider 不能分配 record / turn / event id。
- Player 可见的揭示内容只来自新的 rumor / visible event，不从 hidden source 派生标题或摘要。
